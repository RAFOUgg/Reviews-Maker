fix(cache): Résout collision cache entre utilisateurs

PROBLÈME:
Tous les utilisateurs voyaient les mêmes statistiques car le système de cache
utilisait des clés partagées ('userStats', 'discordInfo') sans distinction par
utilisateur. Cela causait:
- Fuite de données entre comptes
- Statistiques incorrectes affichées
- Violation de confidentialité

ROOT CAUSE:
UserDataManager.getUserStats() et getUserProfile() utilisaient des clés de cache
fixes identiques pour tous les utilisateurs, créant des collisions:
  User A login → cache['userStats'] = stats A
  User B login → lit cache['userStats'] → voit stats A ❌

SOLUTION:
1. Clés de cache uniques par email:
   - 'userStats' → 'userStats_email@example.com'
   - 'discordInfo' → 'discordInfo_email@example.com'

2. Migration automatique des caches legacy:
   - Ajout de UserDataManager.clearLegacyCache()
   - Appelé au démarrage dans initDatabase()
   - Supprime les anciennes clés partagées

3. Invalidation du cache à la déconnexion:
   - Ajout de UserDataManager.invalidateUserCache(email)
   - Appelé dans les 2 event listeners de déconnexion
   - Garantit pas de persistance entre sessions

CHANGEMENTS:
- app.js (ligne ~2209): getUserProfile() utilise cacheKey unique
- app.js (ligne ~2269): getUserStats() utilise cacheKey unique
- app.js (ligne ~2339): Ajout méthode invalidateUserCache()
- app.js (ligne ~2350): Ajout méthode clearLegacyCache()
- app.js (ligne ~2763): Appel clearLegacyCache() au démarrage
- app.js (ligne ~56): Invalidation cache lors déconnexion (modal account)
- app.js (ligne ~2114): Invalidation cache lors déconnexion (modal auth)

TESTS MANUELS:
✅ User A se connecte → stats correctes
✅ User B se connecte → stats différentes de A
✅ User A se reconnecte → stats inchangées
✅ Cache legacy supprimé au démarrage
✅ Déconnexion nettoie cache user-specific

IMPACT:
- Aucune migration DB requise
- Aucun breaking change API
- Migration cache transparente pour users
- Performance préservée (TTL toujours actif)
